NAnt, Command Lines, Paths and Spaces

After two long days of cleaning up our huge NAnt scripts to make them more reusable and maintainable, I was counting on being able to pass all paths (e.g. checkout folders, test file folders, deploy folders, etc...) via the command line using NAnt's -D: option. Naturally, I assumed spaces wouldn't be a problem, I mean it is year 2005 after all. Its looking like I was wrong. google didn't find anything encouraging on this either, just one guy saying he map's drive letters to folders to avoid having spaces in his paths. He also mentions he all but stopped using NAnt by using the solution task to just build an existing VS.NET solution. Unfortunately, solution doesn't work with VS2005, so it is not an option for us. I am aware of MSBuild, but it brings it's own complications at this very early stage in it's life.

The reason we need to pass folder locations is because we have multiple build types occurring from the same script (as Phil described, this was a problem for Cruise Control too). And the reason their can be spaces in the folder names, is because... well, everyone has spaces in folders these days. I can't believe this is such an unusual requirement. Anyway, so there's two builds, a "continuous build" that builds every 15 minutes (if there have been check-ins of course), and a "nightly build" that includes some more exhaustive tests, creates more deliverables and so on. I'm also trying to get us away from having to maintain a set of batch files that some of us developers use and maintain on our own dev machines to build the same code. I'd like to encourage the developers to all use and maintain a single build script. To this, the checkout folders, deployment folders, test file folders, compiler configurations (debug, release, etc...), and so on are slightly different, but most of the rest of the script is the same, so I wanted to just set some of the NAnt Properties from the command line.

If does turn out to be the case that I can't use spaces in command line arguments, I'm going to have to go back to the drawing board on this. I just dumped two batch files that did what it is taking 5 NAnt files to accomplish, and the batch files were probably half the size, easier to read, and most definitely easier for other developers to learn. Granted there were a couple fancy tricks in these batch files, but no worse than the angle-bracket mess that you end up with using NAnt. To keep using Cruise Control I think we have to use NAnt (I need to verify that), but I just might end up turning the NAnt scripts into a one liner that calls a batch file.

# re: A solution for reading in properties

Left by Seth Gray at 4/22/2005 3:19 PM

an easy way to read in values for properties is to utilize the <script> element to write a simple c# function to call Console.ReadLine() and then write out the value to the property.
example project below

<?xml version="1.0"?>
<!--///////////////////////////////////////////////////////////////////////////

Readline Test NAnt Project File

Usage: NAnt [clean|scc|build|archive|deploy]

/////////////////////////////////////////////////////////////////////////////-->

<project name="scripttest" default="build">
<target name="test">
<script language="C#" prefix="test" >
<code>
<![CDATA[[Function("ReadLine")]
public static string ReadLine( ) {
return Console.ReadLine();
}
]]>
</code>
</script>

<property name="msg" value="${test::ReadLine()}"/>
<echo message="${msg}" />
</target>
</project>
