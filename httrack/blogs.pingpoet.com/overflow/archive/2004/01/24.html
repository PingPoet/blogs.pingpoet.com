Assembly Version Components

Here is the relevant information from MSDN on the Version class regarding “components” of an assembly’s Version.

----
The format of the version number is as follows. Optional components are shown in square brackets ('[' and ']'):
major.minor[.build[.revision]]
The components are used by convention as follows:
* Major: Assemblies with the same name but different major versions are not interchangeable. This would be appropriate, for example, for a major rewrite of a product where backward compatibility cannot be assumed.
* Minor: If the name and major number on two assemblies are the same, but the minor number is different, this indicates significant enhancement with the intention of backward compatibility. This would be appropriate, for example, on a point release of a product or a fully backward compatible new version of a product.
* Build: A difference in build number represents a recompilation of the same source. This would be appropriate because of processor, platform, or compiler changes.
* Revision: Assemblies with the same name, major, and minor version numbers but different revisions are intended to be fully interchangeable. This would be appropriate to fix a security hole in a previously released assembly.

Subsequent versions of an assembly that differ only by build or revision numbers are considered to be Quick Fix Engineering (QFE) updates of the prior version. If necessary, the build and revision numbers can be honored by changing the version policy in the configuration.
----

What does Microsoft do? They only use three of the components in a version (i.e. they don’t use Revision). The original framework was version 1.0.3705. The version of the framework that shipped with VS2003 and Windows Server 2003, was 1.1.30703. And the upcoming "Whidbey Release" is 1.2.x. In probably a couple years, the "Orcas Release" will be 2.x.x.

For ActiveReports I see two options. Option one is to do exactly what Microsoft does. Basically we would start incrementing the Build component with each drop, and not include a Revision component. Option two is would be to update the Revision number (like we do now) with each drop. When we stabilize every quarter or so and release a significant rollup of fixes (we call this a “Service Pack”), we should update the Build number. Usually, but not always, approximately once per 18 months, (and usually once per major product release) we have a service pack that is a huge update. It has some fixes that required some major internal changes, but we have made a great effort to ensure it is completely backward compatible (there is an ActiveReports for .NET release coming up like this). These updates should have the Minor component updated.

Then each major update to a product that will not necessarily be backward compatible will have it’s Major component incremented. "ActiveReports for .NET Version 2" should have the major version updated. As it will have the same assembly names, and is an evolution on the same original code base (it’s a big step in some areas), but is not "binary backwards compatible".

I am leaning towards option one. When all the developers actually synch up and drop a build to QA, we increment the Build number. Since each of the developers maintain a build on their machine and only synch up about once per week; the developer could use the Revision component to identify different builds that will never leave their machine. Yet the Build component indicates the last synchronized product build they are making changes against.
