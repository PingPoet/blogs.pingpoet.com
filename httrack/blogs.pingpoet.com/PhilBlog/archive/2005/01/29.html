Sourceforge NCover, CruiseControl for .NET, and the XSL file

I spent a great deal of time today trying to get the report from the SourceForge (SF) version of NCover to show up in the CruiseControl for .NET web dashboard. I was able to find several links and references for the NCover version from GotDotNet (GDN) but had no luck finding anything for the SF version. In fact, CruiseControl already has the plugin and XSL file provided from the GDN version, but these do not work with the SF version.

My biggest problem was not really being familiar with XSL, or XSLT, so I had to do some learning. The SF version comes with an XSLT file but this will not work out of the box. In order to make it work I took the XSLT, renamed it to an XSL and converted it to the following. Since I’m not an expert there may be some better ways to do it but this worked for me. Once I created this XSL file and hooked it up to the web dashboard (just replaced the existing NCover.xsl file) the NCover xml report being merged into the project log began showing the way I wanted.

Free therapy, the benefits of task logging.

What I did

The idea of writing out daily logs detailing, to varying extents, the tasks done throughout the day is not a new idea. I've used task logging in school a few times to help me organize my time, I used task logging for a while when I was working for the IS department at college, and at one point I was keeping an hourly log as a type of psychology experiment. There are different reasons for task logging, maybe for personal reasons or maybe because it's required, but with the introduction of wikis and personal blogs, task logging has morphed into the idea of daily blogging and seems to be gaining in popularity.

Recently we were asked to keep a weekly blog on our company's wiki that detailed the task done on a particular project we were working on. After the first phase was completed, it was requested that we expand on this weekly blog so I started doing a daily notes page. I had thought about logging daily tasks before but never did, so with this extra encouragement I was ready to jump right in and get started.

I have seen different ways of putting together these types of daily blogs but I decided I would break my day up into time groups and write out what I did during each time slot. The time groups I created were: P1: Before 10:30, P2: 10:30-Lunch or, P3: >Lunch or 1:00-3:00, P4: 3:00-5:00 and P5: >5:00. From this I created a table on my wiki page (see partial table below).

Week Day Period Tasks
Monday P1 My Tasks...
P2 Tasks...
P3 Tasks...
P4 Tasks...
P5 Tasks...
Tuesday P1 Tasks...
P2 Tasks...
P3 Tasks...
P4 Tasks...
P5 Tasks...

In order to remind myself to update my tasks list at the correct times I found a tool Remind!. I have this tool setup to pop up a message whenever it is time to update the tasks. If I'm at a point where I can update the tasks without interrupting what I'm doing, I'll do it. If I'm in the middle of something I can snooze the task for a while and update the task list when I'm free.

At the beginning of each week I create a list of major task items I need to finish during the week. As I complete, get more information, or need to add more tasks I update the list. The major task items are right above the week's task table and near the top of the page so it's easy to get a quick look when needed.

The wiki is extremely quick and easy to update so I usually only have to spend a few minutes updating a typical task period with what was done. I have not found this to be intrusive at all since half of the update periods are at natural break points: right after lunch and at the end of the day. If at any point I want to expound on something, and don't have the time right then, I can come back later and update it in the table or add it to the situations list (more on this below).

What I've learned

After completing a month of this I've learned quite a bit. I have found this to be extremely therapeutic and very helpful at keeping me on track. When I first started logging out task it was fairly antiseptic but over time it had developed into a more personal log. Since the wiki page is accessible to everyone in the company, and I know a few people read over it, I expound on certain items like I was talking to the person. As a result of this I have started extracting longer item into a situations section right above the table where I write out how I feel about some of the bigger events and situations occurring during the week. I have found this personalization to be the therapeutic part. I write out what I feel and think (keeping in mind the accessibility of course) and can get a lot off my chest. Also, since I know my boss reads through the logs sometimes I can use this as a way to communicate with him since he lives in a different state.

Another effect the daily blogging has had is to keep me busy, not that I'm not normally busy, but I know I'll have to blog about what I did during a specific time period so it prevents me from walking the halls, or wasting time by doing other things to make me look busy. You can only use the "read emails and scanned blogs" a few times before it starts looking suspicious. It also provides the extra incentive to keep up the mundane task needing to get done but that is boring as heck to do.

If keeping a task log is something you want to do, this method is extremely easy and efficient because you're not forced to recall everything that happened during the whole day, nor do you have to spend a lot of time at once to write out the log. This is also a good way to keep track of what's going on during the week as well as keeping important information handy, such as recording a good web site you want to visit later or using it as a pseudo-calendar to help you remember something that needs to be done that week.

I know when it comes to task logging to each his own, but I have found this method to be the best I've ever tried before, not only for the results but in the ease, quickness and non-intrusiveness of it.

How you show a mbUnit report in the CruiseControl for .NET Web Dashboard

I recently needed to update our build process to show our mbUnit reports on the CruiseControl for .NET Web dashboard. Trying to get all the pieces together to figure out how to do this was a scavenger hunt of sorts so I decided to put all the information I gathered together in one place. This is the very basics of what I found and fairly simple to implement right away. I'm sure there are more advanced ways of doing this but hopefully this information will get you going fairly quickly.

  1. Acquire the xsl file
    • Since mbUnit is now being bundled inside the TestDriven.NET project, download the setup from http://www.testdriven.net. The xsl file can be found in the TestDriven.NET install under the following folder: ..\TestDriven.NET 1.0\MbUnit\CruiseControl

  2. Create plugin for the Dashboard
    • Make sure you have the latest source of CrusiseControl for .NET. At the time of this writing the version is 0.8
    • Copy the xsl file to ..\CruiseControl.NET-0.8.source\project\xsl (I renamed it to mbUnit.xsl)
    • Open up the solution file ..\CruiseControl.NET-0.8.source\project\ccnet.sln
    • Navigate to the Webdashboard project's xsl folder and add the xsl file you just copied over
    • Navigate to the Webdashboard project's Plugins folder
    • Add a folder for mbUnit
    • Add a code file to the folder call mbUnitTesResultsPlugin.cs
    • Copy/Paste the following code in to the code file
      using ThoughtWorks.CruiseControl.WebDashboard.Dashboard;
namespace ThoughtWorks.CruiseControl.WebDashboard.Plugins.mbUnit
{
    public class mbUnitTestResultsPlugin : XslReportBuildPlugin
    {
        public mbUnitTestResultsPlugin(IBuildLogTransformer buildLogTransformer)
            : base (buildLogTransformer, @"xsl\mbUnit.xsl", "View mbUnit Report", "ViewmbUnitReport")
        { }
    }
}
    • Compile the project

  3. Deploy
    • Copy the newly compiled Webdashboard assembly to the deployment machine's ..\webdashboard\bin folder
    • copy the xsl file to the deployment machine's ..\webdashboard\xsl folder
    • Added the new plungin and xsl information to the web.config file
    <BUILDPLUGINS>
...
<PLUGIN typeName="ThoughtWorks.CruiseControl.WebDashboard.Plugins.mbUnit.mbUnitTestResultsPlugin" />
</BUILDPLUGINS>
<XSLFILES>
...
    <FILE name="xsl\mbUnit.xsl" />
</XSLFILES>

  4. Putting it all together
    • To make it all work make sure your build script is running mbUnit and generating an xml report
    • In the CruiseControls server config file merge the mbUnit xml report
    <TASKS>
    <MERGE>
        <FILES>
            ...
            <FILE>..\results\mbUnit-report.xml</FILE>
        </FILES>
    </MERGE>
</TASKS>
