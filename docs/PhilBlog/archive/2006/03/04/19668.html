Testing Asp.Net components with TestComplete

Since I just renamed my site from Test Bed I felt it was appropriately ironic to have my first official new post be about a testing tool. I’ve recently had the opportunity to review and recommend for purchase an automated test tool from AutomatedQA called TestComplete . While the scope of the test tool covers a wide range of automated testing my experience has been with creating automated unit tests for an asp.net control.

The tool has proven to be very versatile and allows for scripts to be recorded and edited using common languages. The web testing side of the tool uses the element IDs to find the elements in the DOM which allows for the site to be rearranged and modified without having to redo the scripts.

One area that has impressed me is with the tools ability to automate dynamically changing content. Since the control I’ve been working with uses xmlHttp and some dynamic client side scripts to create elements on the fly the elements are not always known when the page loads. So far TestComplete has been able to handle this with very few problems. When I first started to evaluate the tool I had some issues with the dynamic client side elements but I received a patch from AutomatedQA shortly after sending them a small same of the problem that resolved my issues.

One nice feature the tool has is a DOM/Object browser that lets you drill through all the elements on the page and find what you are looking for. This is very helpful when setting up scripts, as well as being able to grab snippets of dynamically created html that can be used to help debug problems.

The tool has a very capable recorder that seems to work well but I find myself not really using it as much because I like to setup the tests scripts a certain way. The recorder is very helpful though because you can record a quick set of actions and plug in parts of the recording output into a more appropriate script format.

Since the tool uses IDs to find elements the clicking and key inputs are not relative to the screen. Instead they are relative to the specific element. So, for example, if you have a button element you can call a click at location (8, 2) on the element and it will click the button. Now if the site changes and the button moves to another location the script will continue to work as long as the button’s ID doesn’t change.

If you are looking for a tool to help with web site test automation I would recommend checking this tool out. A colleague of mine has been using this tool to test .NET winform projects and he also seems to like the tool. During my time working in QA I’ve evaluated several automated UI testing tools and none of them really filled the needs we had but so far I’ve been very impressed by this tool.
